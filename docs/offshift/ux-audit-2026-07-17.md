# Offshift UX and accessibility audit

Date: 2026-07-17

Scope:

- macOS companion dashboard and protection surface
- exit/off controls
- ChatGPT Apps SDK widget flow
- mascot placement

Evidence used:

- Source review of `offshift_companion/Sources/OffshiftCompanion/CompanionViews.swift`
- Source review of `offshift_companion/Sources/OffshiftCompanion/CompanionStore.swift`
- Source review of `src/offshift/OffshiftWidget.tsx`
- Source review of `src/offshift/offshift.css`
- Source review of `docs/offshift/mascot.md`
- Live macOS screenshot of the current intervention state: `/tmp/offshift_audit/full.png`

Overall verdict:

The control model is clear and the copy is careful about local-only behavior, but the experience is still too text-led, the black protect surface is not yet strong enough for keyboard/low-vision users, and the mascot requirement from the product spec is not actually realized in the UI.

## Highest-priority findings

1. Mascot placement is missing from the actual UI.

   The spec calls for a visible Offshift companion mascot with five states, but the current dashboard and protection windows use only SF Symbols (`moon.stars`, `moon.zzz.fill`) and text. That leaves the experience honest, but visually generic. It also weakens the “companion” personality on the one screen where warmth matters most.

   Source anchors:

   - `offshift_companion/Sources/OffshiftCompanion/CompanionViews.swift:117-120`
   - `offshift_companion/Sources/OffshiftCompanion/CompanionViews.swift:12-15`
   - `docs/offshift/mascot.md`

   Change:

   - Add a compact original mascot asset to the dashboard and the protect surface.
   - Keep it decorative and secondary to the text actions.
   - Make sure the screen is still understandable if the image fails to load.

2. The black protect surface is readable, but the accessibility affordance is too soft.

   The surface is visually strong and the danger/action model is clear, but the smaller explanatory copy is pushed to low-opacity white (`0.55–0.6`) on black and there is no explicit always-visible “leave this screen” control or keyboard hint. Right now the safe exits exist, but they are embedded among other choices instead of being unmistakable.

   Source anchors:

   - `offshift_companion/Sources/OffshiftCompanion/CompanionViews.swift:112-179`
   - `offshift_companion/Sources/OffshiftCompanion/CompanionStore.swift:185-208`

   Change:

   - Increase contrast for the caption-sized helper copy on the black surface.
   - Add one explicit, primary escape path that reads like an exit, not just a break.
   - Make the exit path reachable by keyboard without hunting through the full action set.

3. Dashboard hierarchy is busy, so the main action gets diluted.

   The dashboard has the right ingredients: state, reasons, local-only explanation, mode toggle, pause/off controls, and a separate protection launcher. The problem is that all of them compete in one long vertical stack, and the “Open protection” button sits apart from the core control block. A new user has to parse a lot before they know what to do next.

   Source anchors:

   - `offshift_companion/Sources/OffshiftCompanion/CompanionViews.swift:8-102`

   Change:

   - Promote one clear primary next step at the top.
   - Collapse secondary controls like night-care toggles and developer fixtures behind a lower-priority section.
   - Keep the mode/off controls visible, but visually separate them from the main task.

4. Exit/off labels are safe, but one of them is semantically vague.

   `Pause until tomorrow` and `Turn Offshift off` are concrete. `Take 5` is less so: on the black protect surface it reads like a break suggestion, not a definite way to get out of the interruption. That makes the safest path slightly less discoverable than it should be under stress.

   Source anchors:

   - `offshift_companion/Sources/OffshiftCompanion/CompanionViews.swift:144-200`

   Change:

   - Keep `Take 5`, but pair it with a literal descriptor such as “Leave this screen” or “Start a 5-minute break.”
   - Make the destructive off switch visually and verbally distinct from the recovery actions.

5. The Apps SDK widget flow is structurally good, but the error path is too bare.

   The widget has a good consent model: read-only data first, then app-visible mutations, then refreshed plan state. The loading and success states are also clear. The weak spot is the failure path: it collapses into a refresh-only message, which tells the user the session is invalid but not why the flow stopped or whether any action was unsafe.

   Source anchors:

   - `src/offshift/OffshiftWidget.tsx:60-149`
   - `src/offshift/OffshiftWidget.tsx:165-224`
   - `src/offshift/offshift.css:162-183`

   Change:

   - Add a short stale-session explanation in the error state.
   - Keep the single safe action, but make the “why” explicit.
   - Preserve the existing app-visible mutation boundary and idempotency pattern.

## Prioritized change plan

P1

- Add the Offshift mascot to the dashboard and protection surface.
- Make the protect surface’s smallest text high enough contrast for low-vision use.
- Give the protect surface one obvious escape action that is keyboard-friendly.

P2

- Rework the dashboard hierarchy so the primary next step is visually dominant.
- Re-label or supplement `Take 5` so the exit path is unambiguous.
- Tighten the widget error state copy so stale-session failures are understandable in one read.

P3

- Move developer fixtures farther away from the user-facing dashboard flow in release builds.
- Consider grouping the off/pause controls as a dedicated “leave / silence” cluster.

## Acceptance checks

- Mascot
  - The dashboard shows a real Offshift mascot asset in routine/drift/protect states.
  - The protection surface also shows the mascot without covering the action labels.
  - If the mascot image is hidden or unavailable, the screens remain understandable from text alone.

- Protection surface
  - Keyboard-only users can reach a clear exit action quickly.
  - The weakest helper text on the black surface meets contrast expectations.
  - The safe exit path is visually clearer than the destructive off path.

- Dashboard
  - A first-time user can identify the current state and the next recommended action within a few seconds.
  - Secondary controls do not compete with the main action.
  - Developer fixtures are not part of the normal user flow.

- Widget
  - Loading, connected, success, and error states each say what is happening in plain language.
  - A stale or invalid dashboard session is explained as stale, not just failed.
  - Mutations still require the app-visible click path and keep the existing capability/idempotency boundary.

## What could not be fully checked

- I could not visually verify the dashboard state live because the companion was captured in its black intervention surface during this run.
- I could not verify the live ChatGPT-hosted Apps SDK runtime in this environment; the widget review here is source-based.
- I did not attempt to change product code, and I did not touch any other docs.
